## 1. 本章定位与核心约束

### 1.1 定位
第 2 章不再是原始手册的直接翻译与结构化整理，而是**基于第 1 章全部操作手册素材进行的抽象、整合与逆向推导**。目标是为后续实际软件开发提供可直接参考的：
- 功能架构大图
- 跨模块交互规则
- UI 控件体系规范
- 数据流与接口假设
- 系统级测试标准

### 1.2 核心约束（必须时刻牢记）

| 约束项 | 说明 |
|--------|------|
| **原始资料性质** | CAE IOS 用户操作手册，不是技术白皮书或 API 文档 |
| **可确认的内容** | 界面结构、用户操作步骤、参数范围、状态流转、异常边界 |
| **不可确认的内容** | 真实技术栈、前后端通信协议、数据库结构、真实后端 API 定义 |
| **推导性质** | 本章所有"架构"和"接口"均为**从 UI 现象反向推导的假设**，必须标注置信度 |
| **与第 4 章的关系** | 本章为第 4 章（平台技术架构选型）提供**功能需求输入**，但不预设技术方案 |

#### 3.1.1 核心架构实体定义（SSM / SC）

> ⚠️ **术语基准声明**：以下两个架构实体是后续 3.2 ~ 3.6 全部推导的术语基础，必须在开发启动前确立其边界与归属，避免后续实现时出现模块职责错位。

| 实体 | 全称 | 本质定位 | 开发归属 | 职责边界 |
|------|------|---------|---------|---------|
| **SSM** | Session State Manager（会话状态管理器） | IOS 教员台软件**内部**的逻辑协调层 | **必须由 IOS 教员台自行实现**（高优先级核心模块） | 全局状态仲裁、版本控制（State Version）、批量下发、跨工作区冲突消解（Freeplay / Nested 优先级）、课程生命周期管理、L2 业务校验（如 GW = ZFW + Fuel） |
| **SC** | Simulator Core（模拟器核心） | IOS 教员台所连接的**外部模拟器主机系统（Simulator Host）**的接口抽象 | **非 IOS 开发范畴**，IOS 只负责开发与 SC 对接的**接口适配层**（协议封装、数据转换、网络通信） | 物理仿真（六自由度动力学）、视景渲染（IG）、声音合成、航电 FMS、机场/地形数据库。IOS 通过 SC 向其下发参数（飞机重量、机场选择、故障注入、冻结指令），并接收物理状态反馈（位置、姿态、仪表数据、L3 物理校验结果） |

**关键边界原则**：

1. **SC 不直接面向工作区**：任何工作区不得绕过 SSM 直接调用 SC，以保证版本控制和仲裁逻辑的统一性。
2. **SSM 不替代 SC 的计算**：SSM 只做 L2 业务级校验（格式、范围、等式、依赖顺序），不做 L3 物理级仿真（地形碰撞、起飞可行性、FMS 可计算性）；物理可行性判断必须交由 SC（外部模拟器主机）。
3. **状态变更必须经过 SSM**：即使是看似"直接"的联动（如 Map Slew → CB 位置更新），架构层面也建议经过 SSM 转发，以记录版本与处理冲突。
4. **开发风险警示**：若忽略 SSM 的统一协调职责，让各工作区直接处理状态同步和冲突仲裁，将导致代码高度耦合、状态一致性难以维护、Freeplay / Nested 等高级特性无法实现。SSM 必须在架构设计早期确立，而非在业务代码中被动补全。

**典型交互时序**：教员操作 → CB 封装事件 → SSM L2 校验 → 下发 SC → SC L3 校验 → SC 确认 → SSM 递增 Version → 广播所有相关视图。

8. **录制/回放引擎的存储与索引**：Recording 功能需要支持按时间戳跳转、FMS Memory Save/Recall、飞行状态捕获（FSC）。录制文件的索引密度、随机访问性能、存储配额管理策略是什么？（影响 2.4 数据持久化策略和 2.5 性能/压力测试）

基于 ch1_8 V1.2 大幅扩展后的素材量，建议将 ch2_3 前置到 ch2_2 之前，以利用控件梳理过程建立对 ch1 素材的全面认知：

   - **新增工作量**：数据持久化策略推导（课程/场景/录制/快照/反馈文件的分类管理）

   - **新增工作量**：课前检查自动化验证、状态一致性测试、Footer 全功能测试、系统损坏恢复测试

**总估算：18 ~ 25 天**（按逐小节交付、用户验收的节奏）

### 6.2 与用户协作模式（优化后）

建议采用"**核心论点 + 关键决策点确认**"模式推进，而非"细纲全文逐字确认"：

1. 我完成一个小节的**设计草案细纲**，但交付时只呈现：
   - **核心论点**（该小节要解决哪 3~5 个关键问题）
   - **关键图表草图**（Mermaid 架构图/流程图的核心骨架，非完整美化版）
   - **需要用户决策的 3~5 个关键问题**（如"SSM 是否采用集中式状态管理方案？""SC 接口适配层是否采用统一前缀命名？"）
2. 用户确认核心论点和关键决策后，我生成完整 HTML
3. 用户验收 HTML，提出修改意见
4. 修改完成后，进入下一个小节

这种模式的优点是：
- **大幅降低用户审阅负担**：无需逐字审阅细纲全文，只需关注方向和关键决策
- **避免一次性交付大量内容后发现方向偏差**
- **每个小节的产出可以作为后续小节的输入**，减少返工
- **用户可以及时补充真实技术信息**（如有），降低推测风险
- **加快迭代速度**：细纲确认环节从"全文审阅"缩短为"关键点确认"
