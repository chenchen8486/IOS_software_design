## 1. 本章定位与核心约束

### 1.1 定位
第 2 章不再是原始手册的直接翻译与结构化整理，而是**基于第 1 章全部操作手册素材进行的抽象、整合与逆向推导**。目标是为后续实际软件开发提供可直接参考的：
- 功能架构大图
- 跨模块交互规则
- UI 控件体系规范
- 数据流与接口假设
- 系统级测试标准

### 1.2 核心约束（必须时刻牢记）

| 约束项 | 说明 |
|--------|------|
| **原始资料性质** | CAE IOS 用户操作手册，不是技术白皮书或 API 文档 |
| **可确认的内容** | 界面结构、用户操作步骤、参数范围、状态流转、异常边界 |
| **不可确认的内容** | 真实技术栈、前后端通信协议、数据库结构、真实后端 API 定义 |
| **推导性质** | 本章所有"架构"和"接口"均为**从 UI 现象反向推导的假设**，必须标注置信度 |
| **与第 4 章的关系** | 本章为第 4 章（平台技术架构选型）提供**功能需求输入**，但不预设技术方案 |

## 3. 各小节详细设计

### 3.1 功能架构全景图

**输入来源**：
- ch1_1（六大工作区定义与职责）
- ch1_2_1 ~ ch1_2_7（控制面板各选项卡的数据流推导）
- ch1_5（Lesson Plan Profile 的画布视图与列表视图）
- ch1_6（Map Workspace 的主区域与覆盖层）
- ch1_7（Advanced 工具栏的 9 组按钮）
- ch1_8（RF-01 标准训练 Session 完整流程中的跨工作区依赖）

**输出物**：
- 1 张 Mermaid `graph TD` 功能架构全景图（六大工作区为顶层节点，数据流为边）
- 1 张跨工作区数据依赖矩阵表格（哪些工作区共享同一份数据）
- 1 张状态同步规则表格（数据变更后的传播路径）

**关键设计决策待确认**：
- 图中是否引入"模拟器核心"作为中心节点？（建议引入，因为所有工作区最终都与之交互）
- 是否区分"只读数据流"和"可写数据流"？（建议用不同颜色/线型区分）

### 3.2 端到端交互完整性流程

**输入来源**：
- ch1_8（RF-01 ~ RF-06 全部参考流程）
- 各 HTML 的「三、全局操作流程与推导」章节

**输出物**：
- 1 张超大 Mermaid `flowchart LR` 端到端流程图（可作为本章核心图）
- 1 张跨工作区状态联动矩阵表格（行：操作源；列：受影响工作区；单元格：预期行为）
- 1 张异常恢复决策树（Mermaid `flowchart TD`）
- 1 张并发冲突规则表格（参数名、冲突场景、优先级规则、预期结果）

**关键设计决策待确认**：
- 端到端流程图粒度：是画到"点击按钮"级别，还是画到"执行步骤"级别？（建议后者，与 ch1_8 同级）
- 并发冲突规则的范围：是只覆盖已观察到的冲突（如 Weight & CG 的联动），还是推演潜在冲突？

### 3.3 UI 控件体系规范

**输入来源**：
- ch1_1（2.1.2 用户交互控件完整定义）
- 各 HTML 中大量具体控件的使用案例（如 Aircraft Tab 的 Fuel Keypad、Current Conditions 的滚轮选择器）

**输出物**：
- 1 张基础控件库总表（控件名、英文、类型、状态定义、交互规则、使用场景、典型参数）
- 每个控件的 Mermaid `stateDiagram-v2` 状态机（如果状态较复杂）
- 1 张参数输入范式统一规范表（参数类型 → 推荐控件 → 确认方式 → 取消方式）
- 1 张跨工作区控件复用映射表（控件名 → 出现在哪些 HTML → 行为是否一致 → 不一致时备注）

**关键设计决策待确认**：
- 是否将 Sliding Panel 内部的子控件也纳入规范？（建议纳入，因为大量功能在 Sliding Panel 内）
- 控件复用映射中，若发现同一控件在不同工作区行为不一致，是否需要标记为"设计缺陷"或"合理差异"？

### 3.4 数据流与接口契约推导

**输入来源**：
- 各 HTML 的「3.4 数据流与开发流程推导」
- 各 HTML 的「3.4.2 API 接口契约建议」表格

**输出物**：
- 1 张全量数据流汇总图（Mermaid `flowchart LR` 或文字版，按模块分组）
- 1 张接口推测总表（接口名、方法、请求字段、响应字段、来源 HTML、置信度、备注）
- 1 张隐含接口补全表（当前 HTML 未显式推导但跨模块通信必需的接口）
- 1 张已知盲区清单（技术问题、无法推导的内容、需要真实技术输入的项）

**关键设计决策待确认**：
- 接口命名规范：是保持各 HTML 已有的命名，还是统一重命名？（建议统一前缀，如 `ios.lessonplan.executeStep`）
- 置信度定义：
  - **高**：从手册可直接确认的行为（如 `getLessonDetail` 返回课程阶段列表）
  - **中**：基于多个手册章节交叉推断的行为（如 `executeStep` 的 `nextStepId` 返回）
  - **低**：纯假设，无直接手册支撑（如错误码体系的具体编号）

### 5.1 当前已识别的风险

| 编号 | 风险描述 | 影响章节 | 缓解措施 |

### 5.2 对第 4 章的技术输入要求

第 4 章（平台技术架构选型）需要回答以下问题，才能使第 2 章中的推测落地为确定设计：

1. **前端技术栈**：HTML/JS/CSS？Qt/QML？WPF？Electron？（直接影响 2.3 UI 组件规范的可实现性）
2. **前后端通信协议**：REST？WebSocket？gRPC？共享内存？（直接影响 2.4 接口契约的格式定义）
3. **状态管理模式**：集中式状态管理（Redux/Vuex）还是各工作区独立状态？（直接影响 2.1 数据流向和 2.2 联动规则）
4. **模拟器核心接口**：CAE 是否提供官方 SDK 或 API 文档？（直接影响 2.4 接口推测的准确性）
5. **数据持久化策略**：课程文件、场景文件、录制文件的存储格式和位置？（影响 2.2 流程中的加载/保存节点）

## 6. 执行计划与工作量评估

### 6.1 建议执行顺序

1. **ch2_1 功能架构全景图**（2 ~ 3 天）
   - 需要先通读所有 ch1 HTML 的全局流程图和数据流推导
   - 产出 architecture diagram 是最基础的输入，后续章节均依赖它

2. **ch2_2 端到端交互完整性流程**（3 ~ 4 天）
   - 基于 ch2_1 的架构图和 ch1_8 的参考流程进行整合
   - 跨工作区联动矩阵需要仔细核对每个 HTML 的关联依赖表格

3. **ch2_3 UI 控件体系规范**（2 ~ 3 天）
   - 主要基于 ch1_1 和各 HTML 中的控件使用案例
   - 工作量相对可控，但需要对 12 个 HTML 进行全文检索和归类

4. **ch2_4 数据流与接口契约推导**（3 ~ 4 天）
   - 需要汇总 12 个 HTML 的全部接口推测表格
   - 统一命名、去重、补充置信度、补全隐含接口

5. **ch2_5 测试规范与验收标准**（2 ~ 3 天）
   - 基于已有测试要点进行整合和升级
   - 补充系统级和集成级测试

**总估算：12 ~ 17 天**（按逐小节交付、用户验收的节奏）

### 6.2 与用户协作模式

建议采用以下模式推进：

1. 我完成一个小节的**设计草案细纲**（比本章更细的逐段内容规划）
2. 用户确认细纲后，我生成对应 HTML
3. 用户验收 HTML，提出修改意见
4. 修改完成后，进入下一个小节

这种模式的优点是：
- 避免一次性交付大量内容后发现方向偏差
- 每个小节的产出可以作为后续小节的输入，减少返工
- 用户可以及时补充真实技术信息（如有），降低推测风险
