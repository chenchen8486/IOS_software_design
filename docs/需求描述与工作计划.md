### 1.3 多文档对照逻辑与使用方式

本项目同时维护多份来源、不同层级、不同用途的文档，其配合方式如下：

| 文档角色 | 作用 |
|----------|------|
| **A320.md（入口文档）** | 用户基于原始资料翻译、拆解、整理后的**中文主文档**，是后续细化整理与 HTML 可视化的**直接输入源**。 |
| **CAE IOS使用手册.md / CAE IOS使用手册_zh.md（原始参考文档）** | PDF 解析后的**原始章节结构**，保留了官方手册的章节编号（如 `3.1`、`4.2.3`、`7.5.1` 等）。由于编号体系完整、层级清晰，**可作为理解功能嵌套关系与界面层级逻辑的权威参考**。 |

**关键说明**：

1. **A320.md 的章节编号是用户自定义的**，与原始参考文档的章节编号**并不一一对应**，但内容百分百一致（均从原始文档拷贝整理而来）。因此，在细化整理过程中，当需要确认某功能点的**精确层级归属**、**上级菜单**或**同级并列关系**时，应以原始参考文档的章节编号体系为准，再映射回 A320.md 的内容进行输出。

2. **原始文档的章节标号是理解嵌套逻辑的重要线索**。例如，原始文档中 `7.5.1 CLEAR AIR Turbulence` 隶属于 `7.5 TURBULENCE Sliding Panel`，而 `7.5` 又隶属于 `7. CURRENT CONDITIONS Tab`，这种数字层级天然反映了软件界面的菜单/面板嵌套关系。助手应充分利用这一点，确保在 HTML 导航树和流程图中所呈现的层级关系与软件实际结构一致。

3. **工作方式**：以 A320.md 为**内容锚点**，以原始参考文档为**结构校验器**。当 A320.md 中某段描述存在歧义或层级不清晰时，立即回溯至原始参考文档进行交叉验证，确保最终输出的 HTML 与开发流程推导准确无误。

## 2. 核心需求

### 2.1 需求一：章节细化整理

针对 `A320.md` 中第 **2 章（软件操作）** 的各个子章节，进行更细粒度的拆解和结构化整理。

**目标**：
- 将每个选项卡/工作区的功能点拆解到三级、四级菜单粒度。
- 明确每个功能点的：
  - **触发方式**（点击/滑动/长按/键盘输入）
  - **输入域**（参数范围、单位、数据类型）
  - **状态流转**（正常态 → 激活态 → 结果态）
  - **关联依赖**（与其他功能/面板的联动关系）
  - **异常边界**（禁用条件、越界处理、冻结状态影响）

### 2.2 需求二：HTML 可视化输出

将细化后的章节内容，按以下标准转化为 HTML 文件：

| 要求 | 说明 |
|------|------|
| **多文件拆分** | 每个一级/二级章节独立为一个 HTML 文件，便于分章节浏览 |
| **导航体系** | 每个 HTML 包含左侧章节导航树（可折叠）、顶部面包屑、章节间上一页/下一页 |
| **表格美化** | 功能对照表、参数表采用响应式表格，支持表头固定、行高亮 |
| **层级折叠** | 三级及以下内容默认折叠，点击展开，保持页面清爽 |
| **流程图** | 对涉及状态流转、操作步骤的功能，使用 Mermaid.js 或流程图组件可视化 |
| **搜索索引** | 提供全局搜索入口，支持按功能名、参数名、控件类型检索 |
| **中英对照** | 关键术语保留英文原文，如：课程计划（Lesson Plan）、滑动面板（Sliding Panel） |

**HTML 输出目录**：`output/html/`  
**命名规范**：`ch<章节号>_<章节英文名>.html`，例如：`ch2_2_control_board.html`

### 2.3 需求三：逆向开发流程推导

基于细化后的功能规格，为后续待填充的章节（第 4~10 章）提供输入，**逆向推导**出每个功能的开发流程：

| 推导维度 | 内容 |
|----------|------|
| **数据流** | 教员操作 → UI 事件 → 状态变更 → 模拟器指令 → 反馈呈现 |
| **状态机** | 每个功能模块的状态定义、状态转换条件、守卫条件 |
| **API 契约** | 前后端交互的接口名、请求/响应字段、错误码 |
| **组件映射** | 功能点对应到 UI 组件库中的具体组件及配置参数 |
| **测试用例** | 正向路径 + 异常路径 + 边界条件的测试要点 |

最终目标：形成一份可指导实际开发的**软件功能开发手册**。

## 3. 约束与规范

- **语言**：全部使用中文，专业术语首次出现时附英文原文。
- **路径规范**：中间产物统一放入 `temp/`，最终 HTML 放入 `output/html/`，文档放入 `docs/`。
- **零硬编码**：HTML 模板中的样式、脚本路径、配置项不得硬编码，统一走外部 CSS/JS 或配置注入。
- **跨平台兼容**：文件读写统一使用 `encoding='utf-8-sig'`。

## 4. 工作节奏

采用**逐章节推进**的方式：

1. **用户指定**当前要细化的目标章节（以 A320.md 的章节为准）。
2. **助手进行双文档对照分析**：
   - 以 `A320.md` 为**内容锚点**，提取该章节已整理的功能描述与界面说明。
   - 以 `CAE IOS使用手册.md` / `CAE IOS使用手册_zh.md` 为**结构校验器**，利用原始文档的章节编号体系确认功能点的**精确层级归属**、**上级菜单**与**同级并列关系**，确保嵌套逻辑与软件实际结构一致。
   - 当 A320.md 中某段描述存在歧义或层级不清晰时，立即回溯至原始参考文档进行交叉验证。
3. **输出该章节的结构化 Markdown + HTML 可视化文件**。
4. **同步提炼该章节对应的开发流程要点**，填入第 4~10 章的对应位置。
5. **用户验收后**，进入下一章节。
