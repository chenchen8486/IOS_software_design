2.2.4 RF-04 故障注入与 RF-05 重新定位交互完整性

推导来源：RF-04 故障注入与应急训练 + RF-05 地图监控与重新定位  |  输入约束：2.1.3 数据依赖矩阵（故障状态 / 飞机物理状态域）、2.1.4 原则一（全局原子状态标志）  |  输出产物：GAP-05 GAP-06

RF-04（故障注入）和 RF-05（重新定位）都是训练进行中的动态干预操作，它们打破了 RF-02 的常规执行节奏，对状态一致性有极高的实时性要求。本节推导它们的交互序列，并识别与冻结、课程执行等状态的组合约束。

GAP-05：预位故障触发条件评估的归属中优先级
ch1_8 描述预位故障等待"预设触发条件（如高度、速度、时间）"，但 2.1 架构中未明确触发条件评估引擎的归属。

架构补充建议：触发条件评估建议由 Simulator Core 内部的专用触发器引擎执行，而非 SSM。原因：触发条件通常与物理状态（高度、速度、位置）强耦合，由物理引擎内部评估可避免跨进程通信延迟。当触发条件满足时，触发器引擎向 SSM 发送"Armed → Active"事件，SSM 负责后续的广播同步。

级联 Reset 的架构原则：故障解除 ≠ 系统恢复。架构上必须将"故障状态移除"和"系统健康恢复"视为两个独立阶段，中间由强制健康检查连接。这避免了教员误以为"解除故障就万事大吉"，而实际上液压压力或发动机 EGT 仍处于异常区间。

2D/3D 视图状态：纯渲染层状态

RF-05 中的 2D/3D 视图切换、缩放级别、覆盖层选择等操作，从架构视角看属于纯渲染层状态，不影响模拟器核心的物理计算，也不应进入全局 Session State。

架构推导结论：Map 的视图状态（2D/3D、缩放、覆盖层开关）应作为 Map 工作区的本地状态管理，理由如下：
  • 视图切换不改变模拟器核心状态，无跨工作区同步需求
  • 教员可能使用多个显示设备，每个设备的 Map 视图独立（如主屏 3D、副屏 2D）
  • 视图状态纳入全局 Session State 会增加不必要的状态版本开销
例外：如果视图状态需要被 Recording 捕获并在回放时恢复（如"教员当时在看 3D 进近视图"），则应在录制数据包中作为附属元数据保存，但不纳入核心状态管理。

边界约束：FMS Memory Save/Recall 不触发 State Version 递增，也不影响飞机位置/姿态/环境参数。因为它操作的是 FMS 子模块内部的飞行计划数据，而非全局 Session State。这与 Flight State Capture（FSC）有本质区别——FSC 保存的是全量状态，Recall 时必须遵循 2.1.4 原则四的 8 步恢复顺序。

交互断点识别（RF-04/05）

GAP-06：重新定位操作互斥锁高优先级
ch1_8 6.2 节提到"冻结后调整"是一种重新定位方式，但未明确说明冻结期间是否允许其他全局状态变更（如课程执行推进、故障注入、环境参数修改）。

架构补充建议：Session State Manager 需维护一个操作互斥锁（Operation Mutex），规则如下：
  • 重新定位操作期间（从请求到确认），禁止其他工作区修改飞机位置/姿态/航向
  • 冻结期间允许重新定位，但重新定位完成后必须再次确认才能解冻
  • 课程执行引擎在重新定位期间可继续运行（如 Autorun 步骤计时），但 SIM 步骤的物理效果应延迟至重新定位确认后生效
