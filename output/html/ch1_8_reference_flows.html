1.8 参考流程 Reference Flows

整理时间：2026-06-08  |  性质：基于第 1 章各工作区操作手册整合的参考流程  |  细化粒度：端到端流程串联 + 跨工作区联动

本章定位：本章不是对原始手册某一章节的直接翻译，而是基于 1.1 ~ 1.7 全部工作区操作手册中的功能定义、状态机与数据流推导，进行跨章节的流程整合与再创作。目标是为教员提供从训练准备到结束讲评的端到端标准操作参考（SOP），同时为后续第 2 章软件架构设计提供跨模块交互的输入。

一、概述

1.1 参考流程的设计原则

IOS 教员台的 6 大工作区（控制面板、地图、课程计划配置、仪表复示器、维护、高级）在功能上是相互独立的，但在实际训练 session 中，教员需要跨工作区频繁切换和操作。本章将分散在各工作区手册中的局部流程，按照真实训练的时间线重新串联，形成以下 6 类参考流程：

  • 每个流程均以 Mermaid 流程图呈现主干路径，关键决策点用菱形节点标注
  • 流程中的每个操作步骤均可在 1.1 ~ 1.7 对应 HTML 中找到详细的功能说明、参数定义和边界行为
  • 流程之间的衔接点（即一个流程的结束是另一个流程的开始）用虚线标注

3.2 课程执行中的教员干预点

无论 Autorun 还是 Step-by-Step 模式，教员在课程执行过程中均有以下实时干预能力：

  • 切换视图：随时切换至 Map Workspace 或 Lesson Plan Profile，查看飞机位置与课程进度，不影响课程执行
  • 点击 Go to Current：在 LESSON PLAN Tab 中快速定位到当前正在执行的步骤
  • 查阅文档：切换至 DOC Tab 查阅航图、检查单、SOP 或训练大纲；文档查阅不中断课程执行，可随时返回
  • 注入故障：通过 Advanced → Failure 或 AIRCRAFT Tab 的 Malfunctions 按钮激活/预位故障
  • 修改环境：通过 Advanced → Environment 修改天气、时间、灯光等参数（可能影响当前步骤的模拟条件）
  • 冻结模拟器：通过 Footer 的 Freeze 按钮暂停训练，解冻后课程可继续

3.5 UPRT（异常姿态预防与恢复训练）特别说明

UPRT（Upset Prevention and Recovery Training）是独立于常规 Malfunction 故障注入的特殊训练类型，通过 Advanced → UPRT 专门入口执行。其特点如下：

  • 场景驱动：UPRT 提供基于机动（Maneuver Based）和基于场景（Scenario Based）两类训练脚本，教员可在 Data Summary 面板中实时监控关键参数（迎角、坡度、空速等）
  • 课程配合：UPRT 训练通常需要配合专门的 UPRT 课程计划使用，常规课程的 MALF 步骤不会自动触发 UPRT 场景
  • 与 RF-04 的关系：如需在常规训练中临时触发 UPRT 场景，应跳转至 Advanced → UPRT，而非 AIRCRAFT Tab 的 Malfunction Section
安全提示：UPRT 场景可能触发大坡度、大迎角或失速状态，建议在执行前确认 Motion 和 Control Loading 系统已就绪，并提前告知学员训练性质。
架构原则映射（基于 1.1 节）：
• 同一模型，多重视图：LESSON PLAN Tab 与 Lesson Plan Profile 是同一课程数据的多重视图，RF-02 的核心价值在于教会教员根据认知目标（细节 vs 全局）选择合适视图。
• 应急通道设计：Freeplay 和 Nested 跳转打破了常规步骤序列，为教员提供了「绕过当前课程剧本」的直达能力，这与 Advanced 的应急哲学一致。

4.2 初始化顺序建议

上述流程图展示的是推荐顺序，实际操作中教员可根据场景灵活调整。但以下依赖关系不可颠倒：

  • 必须先设机场，再设跑道：REFERENCE AIRPORT Tab 中的跑道列表依赖于所选机场
  • 必须先设跑道，再设跑道条件：CURRENT CONDITIONS Tab 中的跑道条件（Runway Condition）与具体跑道绑定
  • Weight & CG 的联动：修改 GW 后，GW CG 可能自动变化；修改 ZFW 后，Fuel 可能需重新校准以确保 GW = ZFW + Fuel
  • 环境参数的层次：Surface Wind 是 QNH 计算的基础输入之一；温度与 ISA 偏差影响气压高度修正

4.3 场景保存与复用（Save/Recall Scenario）

如需在多次训练中重复使用相同的环境配置（例如标准化 CAT II 进近训练、或特定风切变条件），教员可在 Advanced → Environment → Weather Region 中使用「保存场景（Save Scenario）」和「调用场景（Recall Scenario）」功能：

  • 保存范围：海平面温度、中间层/对流层顶温度、QNH、地面/中间层/对流层顶风、云和能见度（注意：活跃风暴、微下击暴流、风切变、结冰条件不被保存）
  • 跨地形调用：从不同地形海拔位置调用时，地面温度和 QFE 会根据保留数据重新计算，以保持相同的大气剖面
  • 管理操作：已保存场景支持加载（Load）、另存为（Save As）和删除（Delete）
效率提示：对于每周重复的标准化训练（如周一 CAT I、周三 CAT II、周五 Low Visibility），建议预先在 Advanced → Environment 中保存 3~5 个常用场景，每次训练前一键调用，可节省 3~5 分钟环境配置时间。
架构原则映射（基于 1.1 节）：
• 按认知模态分离：RF-03 是纯逻辑推理流程——每个步骤都是精确的数值输入与包线验证，不涉及空间直觉（Map）或流程监控（Lesson Plan）。
• 同一模型，多重视图：环境参数既可在 CURRENT CONDITIONS Tab 中逐步配置，也可通过 Advanced → Environment 的 Save/Recall 快速切换，后者是前者的「快捷方式视图」。
• 按使用频率分层：Save/Recall Scenario 属于「低频但省时」功能，被放在 Advanced 中而非 Control Board 主界面，避免了日常界面的认知噪声。

6.4 FMS 内存保存与召回（FMS Memory Save/Recall）

在进行跨机场重新定位或课程切换前，如需保留当前 FMS 飞行计划，可使用 Advanced → Recording → FMS Memory Save：

  • 保存时机：建议在巡航阶段或完成 SID 后保存，避免在进近关键阶段操作
  • 召回时机：重新定位后如需恢复之前的飞行计划，使用 FMS Memory Recall
  • 注意事项：FMS Memory Save/Recall 仅保存飞行计划数据，不保存飞机物理状态或环境参数；如需保存完整状态，应使用 Flight State Capture（FSC）
架构原则映射（基于 1.1 节）：
• 按认知模态分离：RF-05 是纯空间直觉流程——2D/3D 视图切换、缩放、平移、覆盖层选择都激活右脑空间认知，与 Control Board 的数值输入形成正交互补。
• 应急通道设计：Footer Reposition 预设位置是重新定位的「直达电梯」，将 10+ 步操作压缩为 3 次点击，体现了紧急场景下时间效率优先的设计哲学。
• 同一模型，多重视图：空中交通信息既在 Map 上以图标/航迹形式呈现，也在 Advanced → Intruders 中以参数表格形式呈现，教员可根据「态势感知」vs「精确配置」的需求自由切换。

7.5 状态一致性设计说明

冻结、录制、回放是全局性操作，必须保证所有工作区对同一状态达成共识。以下是逆向开发时需要重点实现的一致性约束：

操作一致性约束
冻结（Freeze） 冻结事件由 Footer 或 Advanced 发出后，应由一个全局原子状态管理器接收并更新全局状态标志，随后触发同步事件。所有消费工作区（模拟器核心、Map 渲染引擎、Lesson Plan 执行引擎、Instruments Repeater 数据流）在下一个渲染循环或数据更新周期开始时，必须首先检查该全局标志；若标志为「冻结」，则停止更新自身状态并显示冻结瞬间的快照。关键约束：
  • 模拟器核心暂停物理计算，但 UI 输入事件队列不应清空，允许教员在冻结期间浏览/修改参数
  • Map 中飞机图标、环境动画（风暴、云层）必须同时静止，避免因渲染管线延迟导致"飞机停飞但云还在动"的异常
  • LESSON PLAN 的执行计时器暂停，但当前步骤索引保持不变；解冻后从该步骤继续
  • Instruments Repeater 中所有仪表读数保持在冻结瞬间值，不允许继续接收模拟器新数据
实现策略建议（最终一致性）：对于无法做到严格实时同步的场景（如 Map 渲染与仪表数据流来自不同线程/进程），允许非关键视觉元素（如云层动画的细微变化）存在短暂的（< 100ms）不一致；但所有关键状态（飞机位置、姿态、仪表读数、课程步骤索引）必须在冻结指令发出后的下一个数据更新周期内达成一致。建议在实现时引入一个全局 Session State Version，每次冻结/解冻/调用都递增版本号，各工作区消费状态时校验版本号，以检测和拒绝过期的异步更新。
状态捕捉（Capture） 捕捉操作必须是一个原子快照（Atomic Snapshot），包含以下数据包：
  • 飞机物理状态：位置、姿态、速度、重量与重心、燃油分布
  • 环境状态：参考机场、跑道、地面风、能见度、云层、温度、QNH
  • 课程状态：当前步骤索引、已执行步骤列表、课程计时（可选）
  • 故障状态：激活故障列表、预位故障列表
边界情况：捕捉过程中如飞机正在移动，应捕捉触发瞬间的状态；不应在捕捉过程中修改任何状态。
状态调用（Recall） 调用时应先冻结模拟器，然后按以下顺序恢复状态，以避免中间态被渲染：
  1. 恢复参考机场与跑道（REFERENCE AIRPORT）
  2. 恢复环境参数（CURRENT CONDITIONS）
  3. 恢复飞机物理状态（AIRCRAFT）
  4. 恢复故障状态（Malfunctions）
  5. 恢复课程状态（LESSON PLAN，可选）
  6. 通知 Map 和 Instruments Repeaters 刷新
  7. 解冻并继续训练
失败回滚：如果调用过程中任一状态恢复失败，系统应回滚到调用前的快照，或保持在冻结状态等待教员决策。
录制（Recording） 录制文件应包含时间戳索引，支持按时间戳跳转。录制过程中如触发冻结，录制应继续（记录冻结事件本身），而非暂停；这样回放时才能看到"冻结 → 解冻"的完整过程。
逆向开发警示：冻结/回放功能最容易出现的架构缺陷是"状态同步时序"问题——Map 渲染引擎、Instruments Repeater 数据流、Lesson Plan 执行引擎三者处理消息的速度不同，如果没有统一的状态版本号或全局时间戳，很容易出现"飞机已解冻但地图还在冻结"或"仪表已更新但课程进度没跟上"的不一致现象。建议在实现时引入一个全局 Session State Version，每次冻结/解冻/调用都递增版本号，各工作区消费状态时校验版本号。
架构原则映射（基于 1.1 节）：
• 按使用频率分层：Footer 将高频功能（Freeze、Reset、Sim Controls）常驻底部栏，中频功能（Reposition、Weather Presets、Capture Flight）放在中部，低频功能（Print、Help、Workspace 切换）放在右侧。这种布局符合 Fitts 定律——越常用的按钮越容易触及。
• 应急通道设计：Weather Presets 和 Footer Reposition 都是「绕过详细配置」的直达电梯。教员不需要在 CURRENT CONDITIONS 中逐项调整云高、能见度、跑道条件，只需点一下「CAT II」即可；不需要在 REFERENCE AIRPORT 中选择机场→跑道→停机位，只需点一下「3nm Short Final」即可。
• 按认知模态分离：冻结/回放涉及「时间维度」的认知——教员需要理解「过去的状态（Recording）」、「现在的状态（Freeze）」和「未来的状态（Recall）」。RF-06 通过流程图和状态一致性约束，将这一抽象的时间认知转化为可操作的界面步骤。

8.2 紧急中止流程

当遇到以下情况时，教员应立即执行紧急中止：

  • 模拟器硬件异常（运动平台异响、视景卡死、操纵负荷异常）
  • 座舱内出现真实安全隐患（烟雾、异味、人员受伤）
  • 系统级故障导致 IOS 完全失去对模拟器的控制

紧急中止步骤：

  1. 按下教员台上的硬件急停按钮（如有）或通知飞行员停止操作
  2. 使用 Footer → Reset 或 Freeze 尽可能稳定模拟器状态
  3. 通过 Workspace 切换至 Maintenance 页面，记录故障现象
  4. 通知维护人员，保留当前系统日志和状态
  5. 如需疏散，按模拟机机房应急预案执行

A.1 决策树使用原则

  • 第一步永远是判断"是否需要暂停"：如果情况紧急（学员失能、系统异常、需要立即干预），先按 Footer Freeze，再考虑后续操作。这是"先止血，后治疗"原则。
  • 冻结是最佳参谋时间：冻结期间教员可以慢慢浏览 Advanced、AIRCRAFT Tab 或 Map，不必担心模拟器继续演化；但不要在冻结期间做过多修改，以免解冻后状态跳变让学员困惑。
  • Advanced 不是万能的：如果只是调整燃油重量、修改风速等常规参数，直接打开对应的 Tab（AIRCRAFT / CURRENT CONDITIONS）更高效；Advanced 的优势在于"跨域组合拳"和"紧急直达"。
  • 状态捕捉是保险绳：在注入复杂故障或进行大幅重新定位前，先 Capture 一下当前状态。如果操作结果不理想，可以随时 Recall 回退。
安全红线：当出现运动平台异响、视景卡死、座舱烟雾/异味、人员受伤等涉及物理安全的情况时，不要依赖软件流程，应立即按下教员台上的硬件急停按钮，并按模拟机机房应急预案执行疏散。
