2.2 端到端交互完整性流程

原始手册对应：ch1_8 参考流程（RF-01 ~ RF-06） |  性质：基于操作手册的架构交互推导  |  置信度：中~高

内容性质声明：本章所有内容均基于 1.8 参考流程 中的操作步骤与状态流转，在 2.1 功能架构全景图 定义的约束下进行交互完整性验证与推导。所有"交互断点"与"状态传递契约"均标注了置信度，供后续软件开发参考，不构成最终技术设计约束。

2.2.1 交互完整性分析框架

1.8 参考流程 从操作手册视角回答了"教员在训练中应该如何操作"——它用流程图（Flowchart）描述了 6 个端到端流程的操作步骤、认知模式和效率技巧。而 2.2 节的任务是从软件架构视角回答另一个问题："系统为了支撑这些操作，各工作区之间必须发生怎样的交互？"

这种视角转换的核心价值在于识别交互断点——即操作手册中明确描述了某个行为，但 2.1 的架构推导中尚未为其分配数据流、同步机制或状态管理逻辑。每一个断点都是后续软件实现中必须填补的架构缺口。

分析方法论：四步法

将操作手册流程转化为架构交互模型的标准方法如下：

  1. 提取工作区与数据变更：从 ch1_8 的每个 RF 中，识别出"涉及哪些工作区"、"修改了哪些数据项"、"触发了哪些状态转换"。
  2. 对照 2.1 数据依赖矩阵：将步骤 1 中提取的数据变更，逐一映射到 2.1.3 跨工作区数据依赖矩阵，确认数据所有者、共享范围和同步规则是否已定义。
  3. 推导跨工作区消息序列：使用 Mermaid Sequence Diagram 描述消息的发送方、接收方、方向（请求/响应/广播）、同步属性（同步调用 vs 异步通知）以及数据负载。
  4. 识别交互断点：若某一步操作在 ch1_8 中有描述，但在 2.1 中找不到对应的数据流或同步机制，则标记为交互断点（Interaction Gap），并推导最小可行的架构补充。
约束红线：2.2 节的所有推导必须以 2.1 定义的数据所有权和同步规则为约束。任何与 2.1 矛盾的推导都必须被视为错误并修正，而非推翻 2.1 的结论。

LESSON PLAN Tab ↔ Lesson Plan Profile 双向同步契约推导

ch1_8 的 3.3 节描述了 Tab 与 Profile 的视图选择行为，但并未从架构层面定义它们之间的数据同步契约。基于 2.1.3 的状态同步规则，推导如下：

同步场景触发源消息路径一致性要求
当前步骤索引变更 LESSON PLAN Tab（Execute / Skip / Go to Current） CB -> SSM -> LPP（事件驱动） 强一致性（<100ms）；Profile 必须在 CB 高亮后同步滚动并高亮当前阶段
剖面阶段跳转 Lesson Plan Profile（点击某阶段） LPP -> SSM -> CB（事件驱动） 强一致性（<100ms）；CB 自动定位到对应步骤，但不自动执行
课程加载完成 Session State Manager（课程引擎） SSM -> CB + SSM -> LPP（广播） 强一致性；两者必须同时收到完整的课程结构数据
执行模式切换 教员（Autorun / Step-by-Step） CB -> SSM -> LPP（状态变更） 强一致性；Profile 上的进度标记样式需随模式变化（Autorun 连续流动 vs Step 离散跳跃）
架构推导结论：当前步骤索引的单一事实来源应为 Session State Manager，而非 LESSON PLAN Tab 或 Lesson Plan Profile 中的任意一方。CB 和 LPP 都是该状态的消费视图，任何一方的用户操作（点击 Execute 或点击剖面阶段）都应先发送至 SSM，由 SSM 更新权威状态后再广播至所有视图。这避免了双向绑定导致的竞态条件。

Freeplay / Nested 异常分支的架构补充

ch1_8 的 3.2 节和 3.4 节提到了 Freeplay 和 Nested 两种打破常规步骤序列的异常机制。2.1 架构中未对这两种场景做专门设计，属于明确的交互断点。

GAP-03：Freeplay 优先级仲裁机制高优先级
ch1_8 描述："Freeplay 步骤可能与当前 Autorun 步骤冲突；Freeplay 优先执行，Autorun 在其完成后继续。"

架构补充建议：Session State Manager 需增加步骤优先级仲裁器。规则如下：
  • Freeplay 步骤的优先级高于常规 SIM/MALF 步骤
  • 当 Freeplay 触发时，SSM 暂停当前常规步骤的执行上下文（非冻结，只是挂起）
  • Freeplay 完成后，SSM 恢复被挂起的常规步骤上下文
  • 若 Freeplay 与当前步骤修改了同一参数（如都修改飞机位置），Freeplay 的修改覆盖常规步骤，但需在课程日志中记录覆盖事件
GAP-04：Nested 课程状态栈高优先级
ch1_8 描述："加载子课程并替换当前课程大纲；执行完成后可返回主课程。子课程执行期间，主课程的步骤计时器暂停。"

架构补充建议：SSM 需维护一个课程状态栈（Course State Stack），支持 LIFO 的嵌套调用：
  • 进入 Nested 时：将主课程的 {courseId, stepIndex, stepTimer, executionContext} 压栈
  • 子课程执行期间：栈顶为子课程状态，主课程计时器暂停
  • 子课程完成/中止时：弹出主课程状态，恢复 stepTimer 和 executionContext
  • 最大嵌套深度建议限制为 3 层，防止栈溢出导致状态混乱

参数校验链的架构实现推导

RF-03 的初始化流程揭示了一个三层校验架构，这是 2.1.2 中"UI 层 → 状态管理层 → 模拟器核心层"数据流的具体实例：

三层校验层级定义

L1 UI 层校验
执行者：Control Board 前端控件
校验内容：输入格式（数字/字符串）、取值范围（如 GW 必须在 40t~80t 之间）
失败行为：即时阻止非法输入，给出红色边框/提示文案，不向 SSM 发送请求
L2 业务校验
执行者：Session State Manager
校验内容：跨参数等式（GW = ZFW + Fuel）、包线约束（CG 在 MAC 范围内）、依赖顺序（必须先选机场再选跑道）
失败行为：拒绝下发并回滚到上次合法状态，向 CB 返回具体错误码
L3 物理校验
执行者：Simulator Core
校验内容：物理可行性（飞机在当前机场跑道能否起飞）、地形碰撞检测、FMS 航线可计算性
失败行为：返回物理校验失败，SSM 保持冻结或回滚到上一次通过校验的快照
失败回滚策略：L1 失败无需回滚（仅前端拦截）；L2 失败回滚到 SSM 上一次合法状态；L3 失败回滚到 Simulator Core 上一次通过物理校验的快照。三层校验的粒度逐层收紧，避免了"前端通过但后端崩溃"的状态不一致。

2D/3D 视图状态：纯渲染层状态

RF-05 中的 2D/3D 视图切换、缩放级别、覆盖层选择等操作，从架构视角看属于纯渲染层状态，不影响模拟器核心的物理计算，也不应进入全局 Session State。

架构推导结论：Map 的视图状态（2D/3D、缩放、覆盖层开关）应作为 Map 工作区的本地状态管理，理由如下：
  • 视图切换不改变模拟器核心状态，无跨工作区同步需求
  • 教员可能使用多个显示设备，每个设备的 Map 视图独立（如主屏 3D、副屏 2D）
  • 视图状态纳入全局 Session State 会增加不必要的状态版本开销
例外：如果视图状态需要被 Recording 捕获并在回放时恢复（如"教员当时在看 3D 进近视图"），则应在录制数据包中作为附属元数据保存，但不纳入核心状态管理。

交互断点识别（RF-04/05）

GAP-06：重新定位操作互斥锁高优先级
ch1_8 6.2 节提到"冻结后调整"是一种重新定位方式，但未明确说明冻结期间是否允许其他全局状态变更（如课程执行推进、故障注入、环境参数修改）。

架构补充建议：Session State Manager 需维护一个操作互斥锁（Operation Mutex），规则如下：
  • 重新定位操作期间（从请求到确认），禁止其他工作区修改飞机位置/姿态/航向
  • 冻结期间允许重新定位，但重新定位完成后必须再次确认才能解冻
  • 课程执行引擎在重新定位期间可继续运行（如 Autorun 步骤计时），但 SIM 步骤的物理效果应延迟至重新定位确认后生效

2.2.5 RF-06 冻结、回放与录制交互完整性

推导来源：RF-06 冻结、回放与录制操作流  |  输入约束：2.1.4 状态一致性设计原则（原则一全局原子标志、原则四 8 步恢复顺序）  |  输出产物：GAP-07

RF-06 是 IOS 中影响范围最大的全局操作流程。冻结操作会同时停止所有工作区的状态更新；录制/回放操作涉及全量状态快照的持久化与恢复。本节严格对照 2.1.4 状态一致性设计原则，推导其交互序列与原子性约束。

冻结/解冻交互序列图

冻结是唯一的全局广播型操作——它必须被全部 6 个工作区和模拟器核心同时感知。2.1.4 原则一要求"全局原子状态标志"由单一协调者管理，此处即 Session State Manager。

全局状态快照的原子性约束

录制操作的核心挑战是：在捕获状态快照的过程中，模拟器核心仍在运行，如果不对状态变更加以控制，捕获到的快照将是"中间态"（如飞机位置已更新但环境参数尚未更新），导致回放时出现不一致。

原子性策略实现方式适用场景与权衡
暂停-快照-恢复 录制前触发全局冻结 → 捕获快照 → 保存至存储 → 解冻 推荐方案。实现简单，与现有冻结机制复用；代价是录制瞬间有短暂冻结（约 100~300ms），对训练体验影响极小
复制-写时复制（COW） Simulator Core 维护状态副本，录制引擎读取副本而不暂停主状态流 高复杂度方案。无冻结延迟，但要求物理引擎支持状态副本机制；若引擎为第三方闭源，可能不可行
环形缓冲区回溯 SSM 维护最近 N 秒的状态帧环形缓冲区，录制时回溯到请求时刻的状态帧 折中方案。无冻结延迟，但只能捕获"请求时刻"而非"当前时刻"状态；内存开销与缓冲区大小成正比
架构假设中优先级
推荐采用"暂停-快照-恢复"策略作为默认实现，原因：
  • 与 2.1.4 原则一的"全局原子状态标志"机制完全复用，不引入新架构组件
  • CAE IOS 原始手册中 Recording 操作在 Advanced 页面内，远离主训练界面，暗示录制不是高频实时操作，短暂冻结可接受
  • 若未来需要无冻结录制，可在 COW 或环形缓冲区方案成熟后平滑替换，不影响上层接口

状态恢复顺序的严格验证（对照 2.1.4 原则四）

2.1.4 原则四定义了状态恢复的 8 步顺序约束。本节为 RF-06 的 Recall 流程建立严格的序列图，每步标注"必须完成才能进入下一步"的验证门。

步骤恢复子系统验证门（必须满足的条件）失败行为
1冻结模拟器SC 确认物理引擎已暂停；所有工作区确认冻结若某工作区未响应，强制冻结并记录警告；继续恢复流程但标记该工作区为"未同步"
2恢复参考机场与跑道机场数据库中存在该 ICAO；跑道标识有效；地景加载成功机场数据损坏或缺失 → 回滚到调用前快照，提示"机场数据异常，请联系维护"
3恢复环境参数所有环境参数在物理包线内；Surface Wind 与 QNH 计算一致参数超限 → 回滚或保持冻结，提示具体超限参数
4恢复飞机物理状态GW = ZFW + Fuel；CG 在包线内；位置不陷入地形等式不成立或地形碰撞 → 回滚，提示"飞机状态非法"
5恢复故障状态所有故障代码在 Malfunction Registry 中存在；冲突检测通过故障代码不存在 → 跳过该故障，记录日志，继续恢复其余状态
6恢复课程状态步骤索引在有效范围；课程定义与当前加载课程一致课程定义不匹配 → 仅恢复飞机/环境/故障状态，课程状态重置为初始
7通知 Map 和 Instruments 刷新Map 渲染完成新场景；IR 显示与物理值一致渲染超时（>500ms）→ 保持冻结，提示"渲染未就绪"
8全参数一致性验证 + 解冻所有子系统状态与快照一致；version 再次递增一致性验证失败 → 回滚到调用前快照或保持冻结等待教员决策
绝不允许"半恢复"状态：上述 8 步中任何一步失败，系统必须回滚到调用前快照或保持冻结。这是 2.1.4 原则四的硬性约束，也是飞行模拟器安全设计的底线——"飞机已恢复但环境未恢复"或"环境已恢复但故障未恢复"都是可能导致训练逻辑崩溃的危险中间态。

GAP-07：录制过程中的动态干预状态处理中优先级

架构补充建议：录制数据流应采用"时间戳事件日志 + 状态快照"的双轨格式：
  • 状态快照：按固定间隔（如 1Hz）捕获全量状态，用于回放时的状态恢复
  • 事件日志：记录离散事件（冻结/解冻、故障注入/解除、重新定位、步骤执行）及其时间戳
  • 回放时，先恢复初始状态快照，再按事件日志的时间顺序重放离散事件，实现"状态 + 动作"的完整复现
