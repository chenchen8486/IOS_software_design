2.2.5 RF-04 故障注入交互完整性

推导来源：RF-04 故障注入与应急训练  |  输入约束：2.1.3 数据依赖矩阵（故障状态 / 飞机物理状态域）、2.1.4 原则一（全局原子状态标志）  |  输出产物：GAP-05

RF-04（故障注入）是训练进行中的动态干预操作，可在课程执行、初始化或冻结后随时触发。本节推导三种注入入口的统一汇聚模型、故障状态机以及系统损坏后的 Reset 级联机制。

前置阅读提示：本节涉及的 SSM（Session State Manager）与 SC（Simulator Core）的详细架构定义参见 2.2.1 核心推导架构实体详解。

ch1_8 的 5.1 节描述了故障的生命周期，但将其映射到 2.1 的架构中，需要明确状态转换的触发者和数据流路径。

状态转换触发者消息路径一致性要求
Idle → Armed教员（三种入口）CB/ADV -> SSM -> Malfunction Registry最终一致性；预位不立即影响物理状态
Armed → Active触发条件引擎 / 教员手动SC 触发器评估 -> SSM -> SC 故障生效 -> SSM 广播强一致性（<50ms）；故障生效后所有工作区必须同步
Active → Reset教员解除 / 课程步骤自动解除CB/ADV -> SSM -> SC 故障解除 -> SSM 广播强一致性；解除后仪表和 Footer 指示灯必须同步熄灭
Active → SysDamageSimulator Core（物理引擎内部）SC -> SSM：系统健康告警强一致性；损坏状态必须立即通知教员
GAP-05：预位故障触发条件评估的归属中优先级
ch1_8 描述预位故障等待"预设触发条件（如高度、速度、时间）"，但 2.1 架构中未明确触发条件评估引擎的归属。

架构补充建议：触发条件评估建议由 Simulator Core 内部的专用触发器引擎执行，而非 SSM。原因：触发条件通常与物理状态（高度、速度、位置）强耦合，由物理引擎内部评估可避免跨进程通信延迟。当触发条件满足时，触发器引擎向 SSM 发送"Armed → Active"事件，SSM 负责后续的广播同步。

级联 Reset 的架构原则：故障解除 ≠ 系统恢复。架构上必须将"故障状态移除"和"系统健康恢复"视为两个独立阶段，中间由强制健康检查连接。这避免了教员误以为"解除故障就万事大吉"，而实际上液压压力或发动机 EGT 仍处于异常区间。
