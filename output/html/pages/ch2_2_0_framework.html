2.2.1 交互完整性分析框架

内容性质声明：本章所有内容均基于 1.8 参考流程（RF-01 ~ RF-06） 中的操作步骤与状态流转，在 2.1 功能架构全景图 定义的约束下进行交互完整性验证与推导。所有"交互断点"与"状态传递契约"均标注了置信度，供后续软件开发参考，不构成最终技术设计约束。

1.8 参考流程 从操作手册视角回答了"教员在训练中应该如何操作"——它用流程图（Flowchart）描述了 6 个端到端流程的操作步骤、认知模式和效率技巧。而 2.2 节的任务是从软件架构视角回答另一个问题："系统为了支撑这些操作，各工作区之间必须发生怎样的交互？"

这种视角转换的核心价值在于识别交互断点——即操作手册中明确描述了某个行为，但 2.1 的架构推导中尚未为其分配数据流、同步机制或状态管理逻辑。每一个断点都是后续软件实现中必须填补的架构缺口。

⚠️ 架构实体来源声明：原始 CAE IOS 手册是操作手册，面向教员描述"如何点击按钮"，不会定义任何软件内部模块。本节出现的 Session State Manager（SSM） 和 Simulator Core（SC） 均为架构推导产物，而非原始文档中的实体——SSM 是为了解释"全局冻结同步、跨面板参数一致、Freeplay 冲突仲裁"这三个现象而假设的逻辑协调层；SC 是对手册中"模拟器/主机/物理计算"的架构抽象命名。两者都是当前阶段的占位符，后续软件实现中的真实模块命名、进程边界乃至职责拆分都可能与之不同。

分析方法论：四步法

将操作手册流程转化为架构交互模型的标准方法如下：

  1. 提取工作区与数据变更：从 ch1_8 的每个 RF 中，识别出"涉及哪些工作区"、"修改了哪些数据项"、"触发了哪些状态转换"。
  2. 对照 2.1 数据依赖矩阵：将步骤 1 中提取的数据变更，逐一映射到 2.1.3 跨工作区数据依赖矩阵，确认数据所有者、共享范围和同步规则是否已定义。
  3. 推导跨工作区消息序列：使用 Mermaid Sequence Diagram 描述消息的发送方、接收方、方向（请求/响应/广播）、同步属性（同步调用 vs 异步通知）以及数据负载。
  4. 识别交互断点：若某一步操作在 ch1_8 中有描述，但在 2.1 中找不到对应的数据流或同步机制，则标记为交互断点（Interaction Gap），并推导最小可行的架构补充。
约束红线：2.2 节的所有推导必须以 2.1 定义的数据所有权和同步规则为约束。任何与 2.1 矛盾的推导都必须被视为错误并修正，而非推翻 2.1 的结论。

Sequence Diagram 参与者命名

为了与 2.1.1 的功能架构全景图保持一致，Mermaid Sequence Diagram 中的参与者（Actor）命名如下：

架构实体与 Sequence Diagram 别名映射

Instructor
INST（教员）
人类操作者，所有交互序列的起点
Control Board
CB（控制面板）
聚合根，承载 AIRCRAFT / CURRENT CONDITIONS / REFERENCE AIRPORT / LESSON PLAN 等 Tabs
Map
MAP（地图）
空间渲染与交互，SLEW 拖动回写
Lesson Plan Profile
LPP（课程计划剖面）
课程数据的可视化剖面视图
Instruments Repeaters
IR（复示仪表）
只读消费端，订阅物理引擎数据
Advanced
ADV（高级）
应急快速通道，直接操作核心或修改 CB 参数
Footer / Instructor Station
FS（底部快捷栏）
全局控制网关，冻结/重置/时间控制/工作区切换
Session State Manager
SSM（状态管理器）
全局状态协调者，仲裁、版本控制、批量下发
Simulator Core
SC（模拟器核心）
物理引擎、环境引擎、声音系统、FMS 子模块
重要约定：在 Sequence Diagram 中，SSM（Session State Manager）被显式画为独立参与者，而非隐式内嵌于某个工作区中。这强调了 2.1.2 中提出的核心假设——SSM 是所有工作区与模拟器核心之间的唯一协调者。所有跨工作区的状态变更（即使是看似"直接"的联动，如 Map Slew → CB 位置更新）在架构层面都建议经过 SSM 转发，以保证版本控制和仲裁逻辑的统一性。
