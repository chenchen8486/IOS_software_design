RF-01 是覆盖训练完整生命周期的主流程，从加载课程到结束讲评，横跨全部 6 大工作区。本节将其转化为跨工作区的消息序列图，并验证每个阶段在 2.1 架构中的覆盖情况。

前置阅读提示：本节序列图大量涉及 SSM（Session State Manager）与 SC（Simulator Core）两个架构实体。若读者对二者的职责边界、管辖范围及协作模式尚不熟悉，建议先阅读 2.2.1 核心推导架构实体详解，再返回本节。

阶段一
课程加载与初始化检查		 阶段目标：从 Lesson Plan 中选定课程，将课程定义加载到 SSM，并把课程结构同步到 Control Board 与 Lesson Plan Profile，完成训练开始前的数据准备。
1Instructor → Control Board进入 LESSON PLAN Tab，选择课程教员在 Control Board 上触发课程选择，是整个训练 Session 的人机交互入口。
2Control Board → Session State Manager课程加载请求 {courseId}CB 只负责收集教员点击，把课程标识（如 courseId = "A320-ILS-APPR-01"）上报给 SSM。由 SSM 而不是 CB 直接访问 SC，是为了让 SSM 成为本次 Session 的单一事实来源：加载完成前其他工作区不会收到半拉子数据；SSM 可统一打上版本号 version+=1；若 SC 失败，SSM 也能统一回滚或报错，而不是让 CB 自己处理。
3Session State Manager → Simulator Core加载课程定义SSM 用 courseId 向 SC 请求完整的课程定义（Course Definition）。它不是一个名字，而是一套结构化数据，至少包括课程元信息、阶段（Phases）、步骤（Steps）、默认参数（Params）和评分/事件规则。
4Simulator Core → Session State Manager课程数据 {phases, steps, params}SC 把课程定义的完整结构返回给 SSM，包括阶段、步骤与默认参数。SSM 拿到这份权威数据后，才会继续分发给 CB（显示大纲）和 LPP（做进度剖面）。如果 CB 越过 SSM 直接找 SC，LPP 就不知道课程有哪些步骤，后续「步骤高亮」和「剖面进度同步」都会断掉。
5Session State Manager → Control Board课程加载完成 {version+=1}SSM 确认加载成功并递增全局版本号，通知 CB 更新界面状态。
6Session State Manager → Lesson Plan Profile同步课程结构（剖面数据）SSM 将课程阶段与步骤列表同步到 LPP，为后续进度高亮做准备。

领域切换点的状态传递契约

RF-01 中存在三个关键的领域切换点（Domain Handoff），即主流程临时进入子流程（RF-03~RF-06）的衔接位置。每个切换点都必须定义清晰的输入/输出状态契约，否则子流程返回后主流程将失去上下文。

切换点源状态（进入子流程前）目标状态（子流程完成后）契约验证规则
A：课程加载 → RF-03 初始化{courseLoaded: true, stepIndex: 0, aircraft: default, env: default, airport: null}{initComplete: true, aircraft: validated, env: validated, airport: validated, version: N}version 必须递增；所有校验链通过；GW = ZFW + Fuel 等式成立
B：实时监控 → RF-04 故障注入{sessionRunning: true, stepIndex: K, malfList: []}{malfList: [...], malfVersion: M, sessionRunning: true}故障状态必须在 50ms 内同步至所有工作区；课程执行状态不丢失
C：实时监控 → RF-05 重新定位{position: P0, heading: H0, flightPhase: F}{position: P1, heading: H1, flightPhase: F, repositionVersion: R}重新定位期间课程执行状态暂停或保持；非着陆接地瞬间允许重新定位
D：任意阶段 ↔ RF-06 冻结/解冻{sessionRunning: true, ...allState}{freezeFlag: true, frozenSnapshot: S, version: V+1}冻结瞬间捕获全量状态快照；所有工作区在 50ms 内响应
{freezeFlag: true, frozenSnapshot: S, version: V+1}{freezeFlag: false, sessionRunning: true, version: V+2}基于 frozenSnapshot 恢复；物理引擎恢复运行；所有工作区在 50ms 内解除冻结

交互断点识别（RF-01）

GAP-01：Debrief 阶段的事件标记机制中优先级
ch1_8 描述教员在讲评阶段需要"标记关键事件（故障注入点/冻结点/学员偏差）"，但 2.1 的数据依赖矩阵中没有定义Event Marker这类数据项。架构补充建议：增加一个轻量级的 Event Marker 类型，包含 {timestamp, type, sourceWorkspace, description, snapshotRef}，由 SSM 在事件发生时自动生成，教员可在 Debrief 阶段审阅和手动补充标注。
GAP-02：Session Cleanup 后的状态验证机制中优先级
ch1_8 描述 Cleanup 阶段需要"复位模拟器状态，准备下一课程"，但 2.1 中未定义Post-Session Validation步骤。架构补充建议：在 Total Reset 或 Session Reset 后，SSM 应自动执行一次全参数一致性校验（飞机是否回默认值、环境是否回 ISA、故障是否清零、冻结是否解除），校验失败时保持冻结并提示教员，避免"半复位"状态进入下一课程。
