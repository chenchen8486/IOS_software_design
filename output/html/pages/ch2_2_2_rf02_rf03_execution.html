LESSON PLAN Tab ↔ Lesson Plan Profile 双向同步契约推导

ch1_8 的 3.3 节描述了 Tab 与 Profile 的视图选择行为，但并未从架构层面定义它们之间的数据同步契约。基于 2.1.3 的状态同步规则，推导如下：

同步场景触发源消息路径一致性要求
当前步骤索引变更 LESSON PLAN Tab（Execute / Skip / Go to Current） CB -> SSM -> LPP（事件驱动） 强一致性（<100ms）；Profile 必须在 CB 高亮后同步滚动并高亮当前阶段
剖面阶段跳转 Lesson Plan Profile（点击某阶段） LPP -> SSM -> CB（事件驱动） 强一致性（<100ms）；CB 自动定位到对应步骤，但不自动执行
课程加载完成 Session State Manager（课程引擎） SSM -> CB + SSM -> LPP（广播） 强一致性；两者必须同时收到完整的课程结构数据
执行模式切换 教员（Autorun / Step-by-Step） CB -> SSM -> LPP（状态变更） 强一致性；Profile 上的进度标记样式需随模式变化（Autorun 连续流动 vs Step 离散跳跃）
架构推导结论：当前步骤索引的单一事实来源应为 Session State Manager，而非 LESSON PLAN Tab 或 Lesson Plan Profile 中的任意一方。CB 和 LPP 都是该状态的消费视图，任何一方的用户操作（点击 Execute 或点击剖面阶段）都应先发送至 SSM，由 SSM 更新权威状态后再广播至所有视图。这避免了双向绑定导致的竞态条件。

Freeplay / Nested 异常分支的架构补充

ch1_8 的 3.2 节和 3.4 节提到了 Freeplay 和 Nested 两种打破常规步骤序列的异常机制。2.1 架构中未对这两种场景做专门设计，属于明确的交互断点。

GAP-03：Freeplay 优先级仲裁机制高优先级
ch1_8 描述："Freeplay 步骤可能与当前 Autorun 步骤冲突；Freeplay 优先执行，Autorun 在其完成后继续。"

架构补充建议：Session State Manager 需增加步骤优先级仲裁器。规则如下：
  • Freeplay 步骤的优先级高于常规 SIM/MALF 步骤
  • 当 Freeplay 触发时，SSM 暂停当前常规步骤的执行上下文（非冻结，只是挂起）
  • Freeplay 完成后，SSM 恢复被挂起的常规步骤上下文
  • 若 Freeplay 与当前步骤修改了同一参数（如都修改飞机位置），Freeplay 的修改覆盖常规步骤，但需在课程日志中记录覆盖事件
GAP-04：Nested 课程状态栈高优先级
ch1_8 描述："加载子课程并替换当前课程大纲；执行完成后可返回主课程。子课程执行期间，主课程的步骤计时器暂停。"

架构补充建议：SSM 需维护一个课程状态栈（Course State Stack），支持 LIFO 的嵌套调用：
  • 进入 Nested 时：将主课程的 {courseId, stepIndex, stepTimer, executionContext} 压栈
  • 子课程执行期间：栈顶为子课程状态，主课程计时器暂停
  • 子课程完成/中止时：弹出主课程状态，恢复 stepTimer 和 executionContext
  • 最大嵌套深度建议限制为 3 层，防止栈溢出导致状态混乱

参数校验链的架构实现推导

RF-03 的初始化流程揭示了一个三层校验架构，这是 2.1.2 中"UI 层 → 状态管理层 → 模拟器核心层"数据流的具体实例：

三层校验层级定义

L1 UI 层校验
执行者：Control Board 前端控件
校验内容：输入格式（数字/字符串）、取值范围（如 GW 必须在 40t~80t 之间）
失败行为：即时阻止非法输入，给出红色边框/提示文案，不向 SSM 发送请求
L2 业务校验
执行者：Session State Manager
校验内容：跨参数等式（GW = ZFW + Fuel）、包线约束（CG 在 MAC 范围内）、依赖顺序（必须先选机场再选跑道）
失败行为：拒绝下发并回滚到上次合法状态，向 CB 返回具体错误码
L3 物理校验
执行者：Simulator Core
校验内容：物理可行性（飞机在当前机场跑道能否起飞）、地形碰撞检测、FMS 航线可计算性
失败行为：返回物理校验失败，SSM 保持冻结或回滚到上一次通过校验的快照
失败回滚策略：L1 失败无需回滚（仅前端拦截）；L2 失败回滚到 SSM 上一次合法状态；L3 失败回滚到 Simulator Core 上一次通过物理校验的快照。三层校验的粒度逐层收紧，避免了"前端通过但后端崩溃"的状态不一致。

Save/Recall Scenario 的状态持久化契约

ch1_8 的 4.3 节描述了 Advanced → Environment 中的 Save/Recall Scenario 功能。将其映射到 2.1 架构中，需要明确保存范围与数据依赖矩阵的对应关系。

场景数据项2.1.3 数据类别是否被 Save/Recall 持久化架构原因
海平面温度、中间层/对流层顶温度环境参数✅ 保存属于教员输入的配置参数，可复用
QNH、QFE环境参数✅ 保存（衍生存储）可由温度和气压剖面重新计算
地面/中间层/对流层顶风环境参数✅ 保存属于教员输入的配置参数

风切变、结冰条件环境参数（瞬时态）❌ 不保存同上，属于触发式瞬时状态

状态持久化契约：Save Scenario 操作的消息序列为：ADV -> SSM -> 持久化存储。SSM 在保存前需过滤掉"瞬时模拟状态"，只持久化"教员输入的配置参数"。Recall 时，SSM 从持久化存储读取配置参数，按 RF-03 的初始化顺序（机场→环境→验证）重新下发至 SC，并递增 State Version 通知所有工作区刷新。
