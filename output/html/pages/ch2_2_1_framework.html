2.2.1 交互完整性分析框架

内容性质声明：本章所有内容均基于 1.8 参考流程（RF-01 ~ RF-06） 中的操作步骤与状态流转，在 2.1 功能架构全景图 定义的约束下进行交互完整性验证与推导。所有"交互断点"与"状态传递契约"均标注了置信度，供后续软件开发参考，不构成最终技术设计约束。

1.8 参考流程 从操作手册视角回答了"教员在训练中应该如何操作"——它用流程图（Flowchart）描述了 6 个端到端流程的操作步骤、认知模式和效率技巧。而 2.2 节的任务是从软件架构视角回答另一个问题："系统为了支撑这些操作，各工作区之间必须发生怎样的交互？"

这种视角转换的核心价值在于识别交互断点——即操作手册中明确描述了某个行为，但 2.1 的架构推导中尚未为其分配数据流、同步机制或状态管理逻辑。每一个断点都是后续软件实现中必须填补的架构缺口。

⚠️ 架构实体来源声明：原始 CAE IOS 手册是操作手册，面向教员描述"如何点击按钮"，不会定义任何软件内部模块。本节出现的 Session State Manager（SSM） 和 Simulator Core（SC） 均为架构推导产物，而非原始文档中的实体。SSM 是 IOS 教员台内部待实现的逻辑协调层；SC 是 IOS 所连接外部模拟器主机系统的接口抽象（非内部模块，亦非硬件）。详见下文「核心推导架构实体详解」。

分析方法论：四步法

将操作手册流程转化为架构交互模型的标准方法如下：

  1. 提取工作区与数据变更：从 ch1_8 的每个 RF 中，识别出"涉及哪些工作区"、"修改了哪些数据项"、"触发了哪些状态转换"。
  2. 对照 2.1 数据依赖矩阵：将步骤 1 中提取的数据变更，逐一映射到 2.1.3 跨工作区数据依赖矩阵，确认数据所有者、共享范围和同步规则是否已定义。
  3. 推导跨工作区消息序列：使用 Mermaid Sequence Diagram 描述消息的发送方、接收方、方向（请求/响应/广播）、同步属性（同步调用 vs 异步通知）以及数据负载。
  4. 识别交互断点：若某一步操作在 ch1_8 中有描述，但在 2.1 中找不到对应的数据流或同步机制，则标记为交互断点（Interaction Gap），并推导最小可行的架构补充。
约束红线：2.2 节的所有推导必须以 2.1 定义的数据所有权和同步规则为约束。任何与 2.1 矛盾的推导都必须被视为错误并修正，而非推翻 2.1 的结论。

核心推导架构实体详解

本节对两个架构推导产物——Session State Manager（SSM）与 Simulator Core（SC）——做专题深入说明。后续 2.2.2 ~ 2.2.7 的所有序列图均默认读者已理解本节定义的模块边界与交互契约。

Session State Manager（SSM）——全局状态协调者

职责定位
所有工作区与模拟器核心之间的唯一协调层，承担状态仲裁、版本控制、批量下发与冲突消解四大职责。SSM 不直接面向教员展示 UI，而是作为"幕后中枢"保证跨工作区数据的一致性。
核心职责
  • 状态仲裁与版本控制：每次状态变更递增 State Version，所有工作区按版本号刷新；管理冻结/解冻、故障激活、重新定位等原子状态标志
  • 课程生命周期：课程加载、步骤推进、执行模式切换（Autorun / Step-by-Step）、终止与重置
  • L2 业务校验与批量下发：跨参数等式验证（如 GW = ZFW + Fuel）、包线约束、依赖顺序校验；校验通过后打包广播至所有相关视图
  • 冲突消解与失败回滚：Freeplay / Nested 优先级裁决、课程状态栈（LIFO）管理；L2 失败回滚至上一次合法状态，L3 失败回滚至 SC 上一次通过校验的快照
  • 持久化契约：Save/Recall Scenario 时过滤"瞬时模拟状态"，仅持久化"教员输入的配置参数"
与 2.1 的对应关系
SSM 是 2.1.2 三层数据流 中状态管理层的具体化，也是 2.1.4 状态一致性原则 中"单一事实来源"机制的执行者。

SSM 的软件形态与内部职责

需要强调的是，SSM 不是「一个大函数」，而是训练 Session 运行期间的唯一管家——一个架构角色 / 状态协调层。它的具体代码形态取决于项目技术栈，但职责边界是固定的。

技术栈风格SSM 的实现形态
桌面端 / Qt / WPF一个 SessionStateManager 单例类，内部持有当前 Session 的状态对象
前端 Web / React / Vue一个全局 Store（Redux / Pinia / Zustand），或一个自定义的 State Machine Hook
后端服务 / 微服务一个独立服务或领域服务（Domain Service），维护 Session 聚合根
事件驱动架构一个事件总线协调器 + 状态投影（Projection），消费 SC 的事件并分发给 UI

无论哪种形态，SSM 都有一个共同特征：持有本次训练 Session 的权威状态副本，并对外提供「读状态 / 申请改状态 / 订阅状态变化」的接口。

从内部看，SSM 可以拆成六个职责模块：

模块职责
状态仓库持有当前 Session 的权威状态：当前课程、步骤索引、飞机状态、环境、故障列表、冻结标志、版本号等
请求路由器接收来自 CB / Map / ADV / Footer 的请求，判断请求类型：加载、启动、注入故障、冻结、结束等
版本与一致性控制每次有效变更递增 State Version；拒绝过期请求；合并并发修改
生命周期管理管理课程状态机：未加载 → 加载中 → 待启动 → 运行中 → 冻结 → 结束 / 回放
冲突解决器裁决 Freeplay / Nested 优先级；处理冻结期间的位置变更、着陆接地瞬间的重新定位等边界冲突
分发器状态变更后，广播给 CB / LPP / Map / IR / Footer 等订阅者
注意：持久化不是 SSM 的必选项。在 IOS 这种实时训练系统里，SSM 主要维护内存中的运行态。真正需要持久化的录像、日志、事件标记，通常由 SC 或专门的 Recording Service 负责。

SSM 与「一个大函数」的关键区别在于：SSM 是有状态、长期存在、职责可横向扩展的。把所有逻辑塞进一个函数会导致「上帝函数」，难以维护版本、生命周期和冲突仲裁。

为什么工作区不能绕过 SSM 直接操作 SC

SSM 保管的是「需要被仲裁、需要保持一致性、需要被版本控制」的会话状态。工作区想和 SC 发生写操作（改状态、改参数、注入故障、冻结等）必须经 SSM，核心原因有三点：

  • 单一事实来源：如果 CB、Map、ADV 都能直接调用 SC 修改状态，多个工作区并发下发不同指令，SC 可能按不同顺序执行，最终状态会乱掉。SSM 作为唯一入口，把所有变更排队、排序、合并，确保 SC 收到的指令是连贯的。
  • 版本控制与一致性：SSM 给每次有效状态变更打一个版本号 State Version，所有工作区都按这个版本号刷新。如果某个工作区因为网络延迟只收到旧版本，SSM 能发现它落后并补齐。没有 SSM，各工作区就无从判断自己看到的是不是最新状态。
  • 冲突裁决与回滚：有些操作不能同时进行，例如冻结期间不允许重新定位、着陆接地瞬间不允许重新定位、课程未加载时不能启动。这些业务规则由 SSM 统一裁决。一旦 SC 执行失败，SSM 可以统一回滚到上一个合法状态。

但 SSM 不是「所有数据的搬运工」。下表区分了哪些交互必须经过 SSM，哪些可以由 SC 直接广播：

交互类型典型操作 / 数据是否必须经 SSM
状态变更（写）课程加载、参数修改、故障注入、重新定位、冻结/解冻必须 经过 SSM。需要仲裁、版本控制、一致性广播与冲突裁决
高频实时数据（读）飞机位置 / 姿态（30Hz）、仪表数据（30Hz）、参数回显不必 经过 SSM。由 SC 直接广播给消费端，避免 SSM 成为性能瓶颈

这样设计的原因很实际：

  • 30Hz 的数据每 33ms 来一次，如果都经过 SSM 转发，SSM 会成为性能瓶颈；
  • 实时数据是「只读测量值」，不是「状态变更请求」，不需要仲裁、不需要版本递增；
  • SSM 只关心「飞机位置是不是被教员手动修改了」，而不关心「飞机当前每一帧的坐标是多少」。
一句话总结：SSM 是「状态变更的守门员 + 版本管理员 + 冲突仲裁员」。工作区向 SC 的写操作必须经 SSM；SC 向工作区推送的高频读数据可以直接广播。

Simulator Core（SC）——外部模拟器主机接口抽象

职责定位
SC 是 IOS 教员台软件的外部边界，代表 IOS 所连接的远端模拟器主机系统（Simulator Host）。它不是 IOS 内部的软件模块，也不是硬件设备，而是架构层面为标识"跨进程/跨网络边界"而引入的接口抽象。所有涉及飞机物理状态变更、环境参数计算、视景驱动、声音合成的请求，都必须通过 SC 离开 IOS 教员台，进入外部模拟器系统执行。
为什么需要 SC
从 IOS 教员台软件的架构视角看，物理仿真和视景渲染是外部能力——IOS 不负责解算六自由度动力学，也不直接渲染三维地景。但 IOS 必须驱动这些外部能力（如设置飞机重量、选择机场、注入故障）并消费其输出（如接收飞机位置、仪表数据、环境状态）。SC 的存在就是为了在序列图中标识："这条消息离开 IOS 边界，进入外部模拟器系统"。
SC 背后的真实系统
  • 物理仿真主机：运行飞行模型（六自由度动力学、气动包线、重量与重心计算、燃油消耗）
  • 视景系统（IG）：负责三维地景渲染、云与能见度、天气现象（风暴、微下击暴流、风切变、结冰）
  • 声音系统：引擎声、告警音、环境音的生成与混合
  • 航电与 FMS：飞行计划计算、导航信号生成、自动驾驶仪逻辑
  • 机场/地形数据库：跑道几何、登机口位置、地形高程
SC 对 IOS 的可见行为
  • 下发接口：IOS 通过 SC 向外部模拟器发送参数变更（飞机重量、机场选择、环境配置、故障注入、冻结指令）
  • 反馈接口：SC 向 IOS 回传参数确认、物理校验结果（L3 校验：地形碰撞、起飞可行性）、状态快照
  • 广播接口：SC 以固定频率（如 30Hz）向 IOS 推送飞机位置/姿态/仪表数据，供 Map、Instruments Repeaters 消费
与 2.1 的对应关系
SC 是 2.1.2 中模拟器核心层的接口抽象，对应数据依赖矩阵中"飞机物理状态"、"环境参数"等强一致性数据项的外部所有者。从 IOS 软件架构看，SC 是必须对接的外部系统边界，而非内部实现单元。

SSM ↔ SC 协作模式

数据流向
下行（SSM → SC）：教员通过工作区发起的参数变更、执行模式切换、冻结/解冻指令、故障注入请求，均经 SSM 校验后下发至 SC。
上行（SC → SSM）：SC 完成参数确认、物理校验结果、步骤自动推进状态、周期广播数据，均回传至 SSM，由 SSM 决定如何分发到各工作区。
边界原则
  • SC 不直接面向工作区：任何工作区不得绕过 SSM 直接调用 SC，以保证版本控制和仲裁逻辑的统一性
  • SSM 不替代 SC 的计算：SSM 只做业务级校验（L2），不做物理仿真（L3）；物理可行性判断必须交由 SC
  • 状态变更必须经过 SSM：即使是看似"直接"的联动（如 Map Slew → CB 位置更新），架构层面也建议经过 SSM 转发，以记录版本与处理冲突
⚠️ 开发归属与实现优先级（必读）：后续 IOS 教员台软件开发必须明确区分 SSM 与 SC 的归属边界，避免资源错配：
  • SSM 是 IOS 教员台内部的核心开发模块：全局状态协调、仲裁、版本控制、批量下发、冲突消解、课程生命周期管理——这些能力不存在于外部系统，必须由 IOS 教员台软件自行实现。SSM 的设计与实现是后续开发阶段的高优先级核心任务，其接口定义直接影响全部 6 大工作区的数据流与状态一致性。
  • SC 是外部模拟器主机系统的接口抽象：IOS 教员台不需要实现物理仿真、视景渲染或声音合成，但需要开发与 SC 对接的接口适配层（协议封装、数据转换、网络通信、异常降级）。SC 的接口规格应以供应商提供的模拟器主机 SDK/协议文档为输入，而非自行推导。
  • 开发风险提醒：若后续开发中忽略 SSM 的统一协调职责，让各工作区直接处理状态同步和冲突仲裁，将导致代码高度耦合、状态一致性难以维护、Freeplay/Nested 等高级特性无法实现。SSM 必须在架构设计早期确立，而非在业务代码中被动补全。
