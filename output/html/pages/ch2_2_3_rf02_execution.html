推导来源：RF-02 课程执行  |  输入约束：2.1.3 数据依赖矩阵（LESSON PLAN / AIRCRAFT / CC / REF AIRPORT 域）  |  输出产物：GAP-03 GAP-04

RF-02 是 RF-01 主流程的核心执行阶段，负责将课程计划（Lesson Plan）中的步骤逐条推进。Autorun 模式下课程引擎自动推进，教员处于"监控者"角色；Step-by-Step 模式下教员的每次点击 Execute 都是一次显式的人机交互事件。本节推导 RF-02 的交互序列，以及 Freeplay / Nested 等异常分支的状态契约。

前置阅读提示：本节序列图涉及的 SSM（Session State Manager）与 SC（Simulator Core）的详细架构定义参见 2.2.1 核心推导架构实体详解。

LESSON PLAN Tab ↔ Lesson Plan Profile 双向同步契约推导

ch1_8 的 3.3 节描述了 Tab 与 Profile 的视图选择行为，但并未从架构层面定义它们之间的数据同步契约。基于 2.1.3 的状态同步规则，推导如下：

同步场景触发源消息路径一致性要求
手动步骤推进 LESSON PLAN Tab（Execute / Skip / Go to Current）或 Lesson Plan Profile（点击阶段） CB / LPP -> SSM -> SC（如需执行）+ CB + LPP（事件驱动） 强一致性（<100ms）；SSM 更新权威索引后广播，Profile 与 CB 同步高亮
自动步骤推进 Simulator Core / Course Execution Engine（Autorun） SC -> SSM -> CB + LPP + MAP/FS（事件驱动） 强一致性（<100ms）；SSM 根据 SC 执行结果推进索引并广播所有视图
课程加载完成 Session State Manager（课程加载流程） SSM -> CB + SSM -> LPP（广播） 强一致性；CB 与 LPP 必须同时收到完整课程结构，SSM 持有初始 currentStepIndex
执行模式切换 教员（Autorun / Step-by-Step） CB -> SSM -> SC + CB + LPP（状态变更） 强一致性；SC 需获取新模式以调整引擎行为，Profile 进度标记样式同步变化
架构推导结论：当前步骤索引的单一事实来源应为 Session State Manager，而非 LESSON PLAN Tab、Lesson Plan Profile 或 Simulator Core 中的任意一方。CB 和 LPP 都是该状态的消费视图，SC 是步骤指令的执行者；任何用户操作（点击 Execute、点击剖面阶段）都应先发送至 SSM，由 SSM 更新权威索引后再广播至所有视图，并视需要向 SC 下发执行指令。这避免了双向绑定导致的竞态条件，也保证了 Autorun 与 Step-by-Step 在状态所有权上的一致性。

Freeplay / Nested 异常分支的架构补充

ch1_8 的 3.2 节和 3.4 节提到了 Freeplay 和 Nested 两种打破常规步骤序列的异常机制。2.1 架构中未对这两种场景做专门设计，属于明确的交互断点。

GAP-03：Freeplay 优先级仲裁机制高优先级
ch1_8 描述："Freeplay 步骤可能与当前 Autorun 步骤冲突；Freeplay 优先执行，Autorun 在其完成后继续。"

架构补充建议：Session State Manager 需增加步骤优先级仲裁器。规则如下：
  • Freeplay 步骤的优先级高于常规 SIM/MALF 步骤
  • 当 Freeplay 触发时，SSM 暂停当前常规步骤的执行上下文（非冻结，只是挂起）
  • Freeplay 完成后，SSM 恢复被挂起的常规步骤上下文
  • 若 Freeplay 与当前步骤修改了同一参数（如都修改飞机位置），Freeplay 的修改覆盖常规步骤，但需在课程日志中记录覆盖事件
GAP-04：Nested 课程状态栈高优先级
ch1_8 描述："加载子课程并替换当前课程大纲；执行完成后可返回主课程。子课程执行期间，主课程的步骤计时器暂停。"

架构补充建议：SSM 需维护一个课程状态栈（Course State Stack），支持 LIFO 的嵌套调用：
  • 进入 Nested 时：将主课程的 {courseId, stepIndex, stepTimer, executionContext} 压栈
  • 子课程执行期间：栈顶为子课程状态，主课程计时器暂停
  • 子课程完成/中止时：弹出主课程状态，恢复 stepTimer 和 executionContext
  • 最大嵌套深度建议限制为 3 层，防止栈溢出导致状态混乱

课程执行边界与降级行为

为保证训练系统在高强度使用中的健壮性，RF-02 的交互设计必须覆盖常见边界与异常路径。以下给出最小可执行的降级协议：

异常场景触发条件协议行为
步骤执行超时 SC 在约定时间内未完成某步骤的模拟动作 SSM 将该步骤标记为 TIMEOUT；通知 CB 高亮并记录课程日志；由教员决定 Skip 或 Retry
Autorun 中强制中断 教员点击 Pause / End Lesson / 冻结 SSM 立即切换会话状态，向 SC 发送停止指令；CB/LPP 同步至中止态
快速连续点击 Execute 教员在步骤尚未完成时多次点击 Execute CB/SSM 层对同一步骤设置状态锁或防抖；重复请求被忽略或返回 BUSY
步骤执行失败 SC 返回执行失败（如 L3 校验不通过） SSM 根据失败类型决策：跳过、重试或终止课程；失败原因写入日志
设计原则：所有边界与降级行为都应先由 SSM 统一决策，再下发给 SC 执行。SSM 是课程执行状态的“仲裁者”，SC 不自行决定课程是否继续或终止。
