对后续章节的输入

对 2.3（UI 控件体系规范）的输入：
  • 冻结期间的 UI 行为：参数输入控件需支持"预调整值"与"当前生效值"的双值显示模式
  • 故障注入确认：三种注入入口的 UI 反馈需统一（激活后红色高亮 + Footer 计数器 + Malfunctions Panel 同步）
  • 重新定位进度指示：Footer Repo 等一键操作需要进度条或"操作中"状态提示，防止教员重复点击
  • Nested 课程切换动画：主课程 → 子课程切换时，Lesson Plan Profile 需有视觉过渡提示，避免教员误以为课程丢失
对 2.4（数据流与接口契约推导）的输入：
  • SSM 接口扩展：需定义步骤优先级仲裁、课程状态栈、操作互斥锁、事件标记的接口契约
  • Malfunction Registry 接口：需定义故障注册、查询、冲突检测、状态转换的 API
  • Recording Engine 接口：需定义状态快照格式、事件日志格式、持久化存储访问接口
  • 8 步恢复顺序接口：需定义每步恢复的输入/输出、验证门、失败回滚的原子性保证
约束重申：2.2 节建立的所有交互序列、状态传递契约和断点清单，是后续 2.3 和 2.4 的顶层约束。任何 UI 控件行为定义或接口契约推导，均不得违背本节定义的数据流方向、状态机转换和异常降级路径。

附录：2.1 架构原则 → 2.2 交互推导映射表

以下表格将 1.1.4 ~ 1.1.11 中总结的五大设计原则，具体化为 2.2 节的交互分析约束条件。

设计原则原则描述对应的第 2.2 章架构决策
原则一：按认知模态分离离散参数与连续空间信息绝不混在同一面板2.2.4 中 Map 的 2D/3D 视图状态被定义为纯渲染层本地状态，不进入全局 Session State；2.2.4 中重新定位的四种方式按"空间直觉"（SLEW/Map）和"逻辑推理"（Airport Repo/Footer Repo）分类分析
原则二：按使用频率分层高频功能常驻主界面；低频功能独立隔离2.2.4 中三种故障注入入口按使用频率分层：MALF（标准化高频）→ Quick（即兴中频）→ Advanced（精细低频）；2.2.5 中 Footer Freeze（高频）与 Advanced Freeze（低频精细控制）的消息路径差异
原则三：按角色隔离不同用户角色的功能绝不混用2.2.4 中故障注入权限与维护锁定状态互斥；2.2.6 异常降级矩阵中，所有无法通过软件恢复的路径最终都降级到 Maintenance 角色域
原则四：同一模型，多重视图同一个领域对象设计多个正交视图2.2.3 中 LESSON PLAN Tab ↔ Lesson Plan Profile 的双向同步契约推导；当前步骤索引的单一事实来源为 SSM，CB 和 LPP 均为消费视图
原则五：应急通道设计常规层级导航在紧急场景下是致命的2.2.4 中 Footer Reposition（3 次点击直达）与 SLEW/Airport Repo（10+ 次操作）的交互代价对比；2.2.5 中 Footer Freeze（瞬时冻结）作为最高频应急操作的消息路径最短化设计
