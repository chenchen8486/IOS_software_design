2.1 功能架构全景图

内容性质声明：本章（第 2 章）所有内容均基于 CAE IOS 操作手册的界面现象、操作步骤与状态流转进行逆向推导，并非 CAE 官方技术白皮书或 API 文档。所有"架构假设"、"接口推测"与"数据流推导"均标注了置信度，供后续软件开发参考，不构成最终技术设计约束。

2.1.1 六大工作区功能分层与依赖关系

基于 1.1 IOS组成部分与设计哲学 中对六大工作区的定义，以及 1.1.4 ~ 1.1.11 的架构抽象逻辑分析，可将 IOS 教员台的功能模块划分为核心训练域（Core Domain）与支撑保障域（Support Domain）两大层级。所有工作区最终均与模拟器核心（Simulator Core）交互，但交互模式（读写、只读、快速干预）存在本质差异。

全局控制子系统：Footer / Instructor Station

在 2.1.1 的功能架构全景图中，有一个未显式画出但贯穿所有状态同步规则的关键模块——Footer（底部快捷栏 / Instructor Station）。根据 1.1.1 的定义，Footer 是 Control Board 三层结构之一（主体面板 + 侧边滑动层 + 底部快捷栏），承担会话级别的全局原子操作：

  • 冻结/解冻（Freeze / Unfreeze）
  • 重置（Position Reset / Total Reset / Session Reset）
  • 时间控制（Speed Up / 模拟时间）
  • 工作区切换（Workspace 按钮）

在架构抽象层面，Footer 虽然是 Control Board 的物理组成部分，但在数据流层面它直接与 Session State Manager 交互，对所有工作区施加全局约束。因此，在理解跨工作区数据依赖和状态同步时，应将 Footer 视为 Control Board 内部的全局控制网关。

核心训练域四模块正交关系

核心训练域的四个模块遵循 "对象-场景-流程-监控" 的正交分解，彼此之间不是层级包含关系，而是通过数据联动保持职责独立：

模块核心关注点抽象逻辑与模拟器核心的交互模式
Control Board 本机实时参数 训练对象——所有与"这架飞机此刻状态"相关的参数，是系统的聚合根 双向读写：修改参数后下发至模拟器核心；核心状态变化实时反馈到面板
Map 空间与环境 训练场景——飞机所处的物理世界（天气、地形、交通、导航台） 双向读写：Slew 拖动飞机位置回写核心；核心推送飞机位置/航迹至地图渲染
Lesson Plan Profile 课程执行时间线 训练剧本——按阶段推进的预定义流程，将训练活动结构化为可编排脚本 间接交互：通过 Control Board 的 LESSON PLAN Tab 向核心下发步骤指令
Instruments Repeaters 座舱仪表复示 观察窗口——教员视角的飞行状态可视化，与学员座舱仪表形成镜像 只读订阅：从模拟器核心接收仪表数据，不反向影响模拟器状态
架构假设高置信度
Instruments Repeaters 被设计为纯消费端，这是从原始手册中 "allows the instructor to select which cockpit instruments to repeat or display" 的描述推导而来——教员只选择"看哪些仪表"，从不通过这些仪表反向控制飞机。这种设计保证了仪表显示不会干扰模拟器核心的物理计算。

模块设计原则功能定位与核心训练域的关系

数据流分层说明

UI 层 → 状态管理层

教员通过 1.1.2 / 1.1.3 中定义的控件体系与系统交互。所有参数修改（无论来自 Control Board 的 Wheel Picker、Map 的 Slew 拖拽，还是 Advanced 的快速按钮）均被抽象为统一的参数变更事件（Parameter Change Event），携带以下元数据：

  • 参数标识符（Parameter ID）
  • 目标值（Target Value）
  • 来源工作区（Source Workspace）
  • 是否需要确认（Require Confirmation）
  • 生效时机（即时 / 冻结后 / 下一帧）

状态管理层 → 模拟器核心层

Session State Manager 是所有工作区与模拟器核心之间的唯一协调者。它的核心职责包括：

  • 状态仲裁：当多个工作区同时修改同一参数时（如 Control Board 和 Advanced 同时修改风速），按"最后写入优先"或"工作区优先级"规则仲裁
  • 版本控制：每次冻结/解冻/状态调用/课程跳转时递增全局 State Version，各工作区消费数据前校验版本号，拒绝过期更新
  • 批量下发：对于状态调用（Recall）等批量恢复场景，按"机场 → 环境 → 飞机 → 故障 → 课程"的顺序原子化下发，避免中间态被渲染
架构假设中置信度
Session State Manager 的存在是从 1.8 中"冻结/录制/回放的跨工作区影响"以及"状态一致性设计"章节推导而来的。原始手册并未明确提及一个"中心状态管理器"，但跨工作区的状态同步行为（如冻结时所有仪表同时静止）暗示了某种集中式协调机制的存在。具体实现可能是集中式状态管理（Redux/Vuex 风格），也可能是基于事件总线的分布式协调——待第 4 章技术选型后确定。

模拟器核心层 → 输出层

模拟器核心完成物理计算后，将结果推送至各消费端：

  • Map 渲染引擎：接收飞机位置、航迹、环境特征（风暴、入侵者）数据
  • Instruments Repeater 数据流：接收姿态、空速、高度、航向等仪表数据
  • Control Board 反馈：当前参数值回显（如 Weight & CG 修改后的新重心位置）
  • 声音系统：引擎声音、风切变警告、TCAS 告警等音频输出

2.1.4 状态一致性设计原则

冻结、回放、状态调用是 IOS 中影响范围最大的三类全局操作。它们必须保证所有工作区对同一状态达成共识，否则将出现"飞机已停飞但仪表还在动"或"Map 已跳转但 Control Board 仍显示旧机场"等严重不一致。本节将 1.8 的 7.5 状态一致性设计 升级为系统级设计原则。

原则一：全局原子状态标志

所有全局性操作（冻结/解冻、状态调用、课程跳转）应由一个全局原子状态管理器接收并更新全局状态标志，随后触发同步事件。各消费工作区在下一个渲染循环或数据更新周期开始时，必须首先检查该全局标志；若标志为"全局变更中"，则停止更新自身状态并等待变更完成通知。

冻结期间的 UI 行为边界：冻结状态下，模拟器核心停止推送新状态，但 UI 输入控件保持可操作。教员可在界面上预调整参数，但参数变更事件在解冻前不会下发至 Session State Manager；UI 层应给出"预调整值"与"当前生效值"的差异提示，避免教员混淆。

原则二：State Version 版本控制

建议在实现时引入一个全局 Session State Version（单调递增的整数或时间戳），每次冻结/解冻/调用/课程跳转都递增版本号。各工作区消费状态时校验版本号，以检测和拒绝过期的异步更新。

设计动机：在 Pub-Sub 广播模式下，各工作区消费数据的时序不同（Map 渲染管线与 Instruments 数据流通常来自不同线程）。若无版本控制，冻结后的旧状态帧可能因网络/线程延迟而迟到，覆盖解冻后的新状态，导致"已解冻但仪表仍显示冻结值"的诡异不一致。State Version 提供了一个单调递增的令牌，使所有消费端能够识别并丢弃过期数据。

原则三：最终一致性策略与允许偏差

对于无法做到严格实时同步的场景（如 Map 渲染与仪表数据流来自不同线程/进程），允许非关键视觉元素存在短暂的（< 100ms）不一致；但所有关键状态必须在操作指令发出后的下一个数据更新周期内达成一致。

架构假设低置信度
上述"State Version 版本控制"机制是从跨工作区状态同步的需求侧推导的假设性设计方案。原始手册并未提及任何版本号机制，但"冻结时所有工作区必须同时静止"这一可观测行为，强烈暗示了某种集中式协调或版本校验的存在。具体实现形式（集中式状态管理器 vs. 分布式事件总线 vs. 共享内存锁）待第 4 章技术选型后确定。

原则四：状态恢复的顺序约束

状态调用（Recall）和重新定位（Reposition）涉及多个子系统的批量更新，必须遵循严格的恢复顺序，以避免中间态被渲染或物理引擎进入非法状态：

  1. 冻结模拟器——防止恢复过程中飞机继续飞行
  2. 恢复参考机场与跑道（REFERENCE AIRPORT）——确定地理基准
  3. 恢复环境参数（CURRENT CONDITIONS）——确定大气基准
  4. 恢复飞机物理状态（AIRCRAFT）——确定对象状态
  5. 恢复故障状态（Malfunctions）——确定系统异常态
  6. 恢复课程状态（LESSON PLAN，可选）——确定流程位置
  7. 通知 Map 和 Instruments Repeaters 刷新
  8. 解冻并继续训练

录制与回放的特殊说明：录制（Recording）与回放（Playback）作为连续时间序列状态操作，其一致性约束比单次状态调用更复杂（涉及时间戳对齐、持续数据帧注入、回放中断时的状态锚定等），具体设计将在后续专门章节中详述。

失败回滚策略：如果调用过程中任一状态恢复失败（如机场数据损坏、环境参数超限），系统应回滚到调用前的快照，或保持在冻结状态等待教员决策。绝不允许"半恢复"状态进入训练流程。

2.1.5 安全边界与权限模型

IOS 教员台不仅是训练工具，也是一台价值数亿元的飞行模拟器的控制中枢。安全边界设计必须保证：维护操作不会污染训练状态，教员的误操作有降级保护，急停信号能够无条件中断一切物理运动。

以下表格将 1.1.4 ~ 1.1.11 中总结的五大设计原则，具体化为第 2 章的架构约束条件。每一条原则都直接影响了 2.1.1 ~ 2.1.5 中的某个关键设计决策。

设计原则原则描述对应的第 2 章架构决策
原则一：按认知模态分离
(Cognitive Modality Separation)		 离散参数与连续空间信息绝不混在同一面板，因为它们激活人脑不同的认知区域 2.1.1 中 Control Board 与 Map 被定义为两个独立工作区，各自拥有独立的渲染管线和数据流；2.1.2 中 UI 控件层将离散参数输入（Wheel Picker / Keypad）与空间交互（Slew / 拖拽）抽象为不同的事件类型
原则二：按使用频率分层
(Frequency-Based Layering)		 高频功能常驻主界面；低频功能独立隔离，减少认知噪声 2.1.1 中核心训练域四模块为高频工作区，支撑保障域两模块为低频工作区；2.1.5 中 Maintenance 通过角色隔离进一步降低教员界面的认知负担
原则三：按角色隔离
(Role-Based Isolation)		 不同用户角色的功能绝不混用，通过独立工作区 + 权限控制实现干净的安全边界 2.1.5 中 Maintenance 被定义为独立操作域，配备 L2 角色层权限校验；急停按钮（L1 物理层）独立于所有软件角色，确保任何角色都能无条件触发
原则四：同一模型，多重视图
(Multiple Views of One Model)		 同一个领域对象值得为其设计多个正交视图，不强迫用户在"看细节"和"看全局"之间做取舍 2.1.3 中课程状态被 Control Board（列表视图）和 Lesson Plan Profile（剖面视图）共享，状态同步规则确保两者始终显示同一套课程数据；2.1.2 中 Session State Manager 作为单一事实来源，向多个视图分发一致的数据
原则五：应急通道设计
(Emergency Bypass Design)		 常规层级导航在紧急场景下是致命的，必须为关键操作预留"直达电梯" 2.1.1 中 Advanced 被定义为 Control Board 的"应急版"，直接操作模拟器核心或修改 Control Board 参数；2.1.5 中急停按钮（E-Stop）是最高级别的应急通道，完全绕过所有软件层级
